1. Contexte 
Votre client, CyberSafe Solutions, est une startup française spécialisée en cyberstabilité 
pour les PME. Avec 150 entreprises clientes et une équipe de 35 développeurs, 
CyberSafe développe des solutions de protection et d’audit pour ses clients. 
Suite à un audit de l’ANSSI (Agence Nationale de la Sécurité des Systèmes d’Information), 
plusieurs failles critiques ont été identifiées dans les projets internes de CyberSafe : 
• Aucun outil centralisé d’analyse de sécurité du code source 
• Vulnérabilités d’injection (SQL, XSS) non détectées avant mise en production 
• Dépendances obsolètes avec des CVE connues dans plusieurs projets 
• Aucune intégration de tests de sécurité dans les pipelines CI/CD 
• Pas de rapport de sécurité automatisé pour les clients 
• Corrections de sécurité manuelles, lentes et sources d’erreurs 
CyberSafe Solutions vous mandate pour développer SecureScan, une plateforme web 
interne d’analyse de qualité et de sécurité de code. Cette plateforme doit orchestrer des 
outils de sécurité open source existants, agréger leurs résultats, les mapper sur l’OWASP 
Top 10 : 2025, et proposer des corrections automatisées.

D. Dashboard de visualisation 
• Score de sécurité global (ex : A/B/C/D/F ou note sur 100)

E. Système de correction automatisé 
Pour les vulnérabilités les plus courantes, l’outil doit proposer des corrections prédéfinies 
(template-based) : 
• Injection SQL → requêtes préparées / paramétrées 
• XSS → échappement des sorties (htmlspecialchars, DOMPurify) 
• Dépendances vulnérables → suggestion de version patchée 
• Secrets exposés → remplacement par variable d’environnement 
• Mots de passe en clair → hachage bcrypt/argon2 
L’utilisateur peut valider ou rejeter chaque correction avant application. 
F. Intégration Git automatisée 
• Création automatique d’une branche de correction (ex : fix/securescan-2026-03-05) 
• Application des corrections validées sur cette branche 
• Push automatique via l’API Git (GitHub API / Octokit ou git CLI) 
• Génération d’un rapport de sécurité (HTML ou PDF) résumant l’analyse et les 
corrections appliquées

2.2 Fonctionnalité BONUS — Correction intelligente par IA 
Cette fonctionnalité est optionnelle et réservée aux groupes les plus avancés. Elle permettra 
de se démarquer lors de l’évaluation. 
Au lieu de corrections template-based uniquement, intégrer un appel à une API LLM 
(OpenAI, Anthropic Claude, ou Mistral) pour : 
• Analyser le contexte complet du code vulnérable 
• Générer une correction adaptée au code existant (pas un template générique) 
• Fournir une explication pédagogique de la vulnérabilité et du fix proposé 
• Permettre à l’utilisateur de valider, modifier ou rejeter la suggestion IA avant 
application 
Exemple d’intégration : envoyer le snippet de code vulnérable + le type de vulnérabilité OWASP à 
l’API, récupérer le code corrigé, l’afficher en diff côte-à-côte dans l’interface. 
3. Technologies 
Le choix des technologies est libre, dans la limite des langages et frameworks étudiés en 
cours. Suggestions : 
• Backend : Node.js (Express), PHP (Symfony/Laravel), Python (Flask/FastAPI) 
• Frontend : React, Vue.js, ou HTML/CSS/JS natif 
• Base de données : MySQL, PostgreSQL, MongoDB, SQLite 
• Outils de sécurité : Semgrep CLI, ESLint + eslint-plugin-security, npm audit, 
Composer audit, git-secrets, TruffleHog 
• Git API : Octokit (GitHub API), GitLab API, ou git CLI (child_process/exec) 
• BONUS IA : API OpenAI, API Anthropic Claude, API Mistral
